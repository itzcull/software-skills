# Style & Formatting

Category 11 -- Does it follow project conventions?

## Subcategories

- **Inconsistent indentation** -- mixing tabs and spaces, or using the wrong indent width
- **Missing/extra whitespace** -- inconsistent spacing around operators, brackets, or keywords
- **Line length violations** -- lines exceeding the project's configured maximum
- **Import ordering** -- imports not sorted according to project conventions (external/internal grouping)
- **Brace style** -- inconsistent use of same-line vs next-line opening braces
- **Trailing whitespace** -- spaces or tabs at the end of lines
- **Inconsistent quoting** -- mixing single and double quotes without a convention
- **Missing trailing commas** -- inconsistent use of trailing commas in multi-line structures
- **Semicolon style** -- inconsistent use or omission of semicolons
- **File organisation** -- inconsistent ordering of exports, types, functions, and constants within a file

## What to Look For

- Formatting that differs from the rest of the files in the diff
- Import blocks that do not follow the project's grouping convention (typically: external libraries, then internal modules, then relative imports)
- Mixed use of `const` arrow functions and `function` declarations without a clear pattern
- Inconsistent blank line usage between function definitions
- Files that do not match the output of the project's configured formatter (Prettier, ESLint, dprint)
- TypeScript files mixing `type` imports with regular imports without using `import type`
- Inconsistent use of explicit return types on functions

## Positive Example

```typescript
import { describe, expect, it } from "vitest";

import { calculateTotal } from "../calculator";
import { createOrder } from "../../test-utils/factories";

describe("calculateTotal", () => {
  it("returns zero for an empty order", () => {
    const order = createOrder({ items: [] });

    const total = calculateTotal(order);

    expect(total).toBe(0);
  });
});
```

Consistent import grouping, blank line separation, and test structure.

## Negative Example

```typescript
import {createOrder} from '../../test-utils/factories'
import {describe, expect, it} from 'vitest'
import { calculateTotal } from "../calculator"

describe('calculateTotal', () => {
  it('returns zero for an empty order', () => {
    const order = createOrder({items: []})
    const total = calculateTotal(order)
    expect(total).toBe(0)
  })
})
```

Mixed quoting styles, inconsistent spacing inside braces, non-standard import order, missing blank lines between arrange/act/assert.

## Common False Positives

- Code that intentionally deviates from formatting rules because the formatter does not handle the case well (annotated with a format-ignore directive)
- Files generated by tools (lock files, codegen output) that should not be manually formatted
- Legacy files that predate the current style guide -- unless the diff touches those lines
- Intentional alignment of related values for readability (column alignment)

## Severity Guide

| Severity | When |
|---|---|
| Critical | Never -- style issues are not critical |
| Major | Rarely -- only if the formatting makes the code genuinely ambiguous or misleading |
| Minor | Inconsistent formatting across the diff that a formatter should catch |
| Nitpick | All other style and formatting preferences |

Style findings should almost always be Nitpick. If a project has an auto-formatter, most style issues should be caught by tooling rather than review. Flag missing formatter configuration rather than individual formatting issues.
